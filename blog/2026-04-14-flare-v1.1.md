# How We Built a Browser-Native LLM Engine That Runs at 224 tok/s

**TL;DR**: Flare is a pure Rust LLM inference engine that compiles to both native binary and WebAssembly from one codebase. In two days of optimization, we went from 19 tok/s to 224 tok/s on a 135M model and reached 38 tok/s on a 1B model — getting within 1.7x of llama.cpp on native, while also running in the browser with zero server costs.

---

## The Problem

Every LLM deployment today involves servers. Whether it's OpenAI's API, a self-hosted vLLM instance, or even "local" solutions like Ollama — they all require running inference on a machine you control. This creates three fundamental problems:

1. **Cost scales linearly**: Every concurrent user needs GPU time
2. **Privacy is impossible**: User data must reach your servers
3. **Latency has a floor**: Network round trips can't go below physics

What if the user's own device could run the model?

## Why Not Just Use llama.cpp?

llama.cpp is excellent for native inference. But it can't run in a browser. Its C/C++ codebase relies on platform-specific backends (Metal, CUDA, OpenCL) that don't exist in WebAssembly. Projects like WebLLM wrap it via Emscripten, but that adds a thick abstraction layer and limits optimization.

Flare takes a different approach: **pure Rust, WASM-first**. One codebase that compiles to native binary (using ARM NEON, AVX2, Apple Accelerate) AND WebAssembly (using WASM SIMD128, WebGPU). No emscripten, no C bindings, no impedance mismatch.

## The Architecture

```
                    ┌─────────────┐
                    │  flare-core │  Tensor, Model, KV Cache, Sampling
                    └──────┬──────┘
              ┌────────────┼────────────┐
              │            │            │
        ┌─────┴─────┐ ┌───┴───┐ ┌──────┴──────┐
        │ flare-gpu  │ │ SIMD  │ │ flare-loader│
        │ (WebGPU)   │ │(WASM) │ │ (GGUF/ST)   │
        └────────────┘ └───────┘ └─────────────┘
              │            │
        ┌─────┴─────┐ ┌───┴───────┐
        │ flare-web  │ │flare-server│
        │ (Browser)  │ │ (Native)   │
        └────────────┘ └───────────┘
```

The key insight: **every layer of the stack is shared**. The same WGSL shader that runs via Metal on macOS runs via WebGPU in Chrome. The same Q8_0 dequantization code runs natively with NEON SIMD or in the browser with WASM SIMD128.

## The Performance Journey

We started at 19 tok/s with naive scalar matmul. Here's how we got to 224 tok/s:

### Day 1: Foundations (19 → 155 tok/s)

| Optimization | Technique | Impact |
|---|---|---|
| ARM NEON SIMD | 4-way float32 accumulator for matvec | 19 → 126 tok/s |
| Pipeline caching | Cache compiled GPU pipelines | +10% |
| GPU-resident forward | Single command encoder, zero readbacks | GPU path working |
| Fused SiLU shader | Combined SiLU + dequant matvec in one dispatch | -1 dispatch/layer |

### Day 2: Closing the Gap (155 → 224 tok/s)

| Optimization | Technique | Impact |
|---|---|---|
| Apple Accelerate | `cblas_sgemv` via AMX coprocessor | 155 → 229 tok/s |
| Q8_0 int8 dot | Quantize input on-the-fly, `sdot` instruction | 1B: 26 → 31 tok/s |
| Fused projections | Q/K/V → single matvec, gate/up → single matvec | 31 → 38 tok/s |
| Online softmax | Flash Attention-style single-pass attention | 187 → 224 tok/s (135M) |
| Zero-alloc forward | Pre-allocated buffers, `_into` functions | -256 allocs/token |
| Greedy fused argmax | Skip 512KB logits buffer for greedy decode | Reduced cache pollution |

### Where We Stand vs. llama.cpp

| Model | Flare | llama.cpp | Gap |
|---|---|---|---|
| SmolLM2-135M (decode) | 224 tok/s | 390 tok/s | 1.7x |
| Llama 3.2 1B (decode) | 38 tok/s | 124 tok/s | 3.2x |

On the 135M model, we're within **1.7x of llama.cpp** — a project with 3+ years of optimization and hundreds of contributors. And unlike llama.cpp, our code also runs in the browser.

## What Makes Flare Different

### vs. llama.cpp
llama.cpp is the gold standard for native inference. But it's C/C++ only, can't target browsers, and requires platform-specific backends. Flare runs the same code everywhere.

### vs. WebLLM / MLC-LLM
WebLLM compiles models through Apache TVM into WebGPU kernels. It's fast in the browser but requires a complex compilation pipeline and model-specific artifacts. Flare loads standard GGUF files directly — the same files you'd use with llama.cpp.

### vs. Transformers.js
Transformers.js uses ONNX Runtime Web for inference. It's excellent for encoder models but not optimized for autoregressive LLM decoding. Flare is purpose-built for token-by-token generation with streaming, speculative decoding, and progressive loading.

### vs. ONNX Runtime Web
ONNX RT Web is a general-purpose runtime. Flare is a specialized LLM engine with quantization-aware kernels, KV cache management, and chat template support built in.

## Features You Won't Find Elsewhere

### Progressive Inference
Start generating text while the model is still downloading. Flare's `forward_partial()` runs inference with only a subset of layers loaded — like progressive JPEG for LLMs. As more layers stream in, output quality improves.

### BitNet 1.58-bit Ternary Weights
Full-stack support for ternary weights ({-1, 0, +1}). Matmul becomes pure addition/subtraction — no multiplication. A 2B model downloads in ~375MB instead of ~1GB. NEON SIMD and WebGPU shader implementations included.

### N-gram Speculative Decoding
Zero-overhead speculation that matches context n-grams against prior output to propose draft tokens. No second model needed — ideal for browser memory constraints.

### KIVI 2-bit KV Cache
Asymmetric 2-bit quantization for KV cache (keys per-channel, values per-token). 8x memory reduction enables much longer context windows within the browser's 4GB WASM limit.

### Multi-Worker WASM
Parallel matmul across Web Workers via SharedArrayBuffer. 3-4x CPU throughput on multi-core devices without WebGPU.

## The Browser Advantage

Running LLMs in the browser isn't just a novelty — it unlocks deployment patterns that are impossible with servers:

**Zero infrastructure**: Ship your model on a CDN. Users download once, cached in OPFS. Your "inference cluster" is every user's device.

**True privacy**: Sensitive queries (medical, legal, financial) never leave the device. There's no server to breach, no logs to subpoena, no data to GDPR-delete.

**Offline capability**: Once cached, models work without internet. Perfect for field workers, airplanes, rural connectivity.

**Economics**: At 100K daily active users with server inference, you're looking at $50K-500K/month in GPU costs. With browser inference, your cost is the CDN bill for model downloads — maybe $500/month.
